# 耳朵only 👂

> 打开即听 · 无推荐 · 无干扰

一个为容易分心的人设计的私人声音播放器。不是内容平台，是声音工具。

---

## 这是什么？

主流音乐App（QQ音乐、网易云、Spotify）的核心逻辑是**用推荐留住用户时长**。但对于一部分人来说（尤其是有ADHD倾向的用户），打开App想听歌 → 被推荐吸引 → 刷歌单 → 忘了自己本来要干嘛。

**耳朵only** 做的是减法：打开就是你的声音，没有推荐流、没有社交、没有排行榜。你来是为了**用**，不是为了**逛**。

## 核心功能

### 1. 场景系统（已确定 ✅）

用户自定义场景，每个场景绑定独立的内容池和界面风格：

| 场景 | 典型内容 | 界面氛围 |
|------|----------|----------|
| 🚗 通勤 | 节奏感强的音乐 | 暖色、能量感 |
| 💻 工作 | Lo-fi / 白噪声 / 纯音乐 | 深色、极简 |
| 🌙 睡前 | 轻音乐 / 播客 / 自然声 | 暗紫、柔和 |
| 🏃 运动 | 高BPM音乐 | 高对比、动感 |
| ☕ 放松 | 爵士 / 民谣 / 播客 | 暖色、舒适 |

场景可由用户自由增删和命名。

### 2. 24小时时间轴（已确定 ✅）

- 一条可视化的24小时时间轴，用户拖拽划分时间段，为每段分配场景
- 例如：`7:00-9:00 通勤（橙）` → `9:00-18:00 工作（青）` → `22:00-23:00 睡前（紫）`
- 直觉化操作，一眼看到自己一天的声音安排
- 长期使用后，AI学习用户习惯，到点自动切换场景（用户仍可随时手动覆盖）

### 3. 双内容源（已确定 ✅）

**本地模式（基础盘）：**
- 用户导入本地音乐文件
- 零版权压力，MVP阶段即可运行
- 支持音乐、播客、白噪声等任意音频文件

**外接模式（增强，实验性）：**
- 对接第三方音乐服务API（网易云、QQ音乐等）
- 参考项目：[NeteaseCloudMusicApi](https://github.com/Binaryify/NeteaseCloudMusicApi) 等开源方案
- ⚠️ 非官方接口，随时可能失效，且商用有法律风险，定位为实验性功能
- 不作为核心依赖

### 4. AI个性化（规划中 🔧）

AI在产品中的角色逐步加深：

| 阶段 | AI能力 | 说明 |
|------|--------|------|
| 前期 | 内容分类 | 用户导入一批歌，AI帮忙分到不同场景 |
| 中期 | 时间感知 | 学习用户时间模式，自动切换场景 |
| 后期 | 多信号融合 | 结合天气、日历、可穿戴设备等数据做智能推荐 |

AI模型：计划使用 Claude Opus 4.6/4.7 或 GPT-5.5 的API，需控制调用频率（非每次播放都触发）。

## 产品原则

- **打开即用**：启动后直接进入当前场景，一键播放
- **零干扰**：没有推荐算法、没有社交功能、没有排行榜、没有广告
- **用户掌控**：所有自动化行为用户都能手动覆盖
- **ADHD友好**：减少决策点，缩短"想听→在听"的路径

## 技术规划

### 技术栈（暂定）

- **前端**：React（当前原型已用React实现）
- **后端**：待定（Node.js / Python）
- **AI接入**：Anthropic API / OpenAI API
- **数据存储**：本地存储 + 云同步（待定）

### MVP路径

```
Phase 1 — 网页版验证核心体验
├── 本地音乐导入与播放
├── 场景配置（增删改）
├── 24小时时间轴交互
└── 极简播放界面

Phase 2 — AI接入
├── 音乐自动分类到场景
├── 时间模式学习与自动切换
└── 用户习惯数据积累

Phase 3 — 多端扩展
├── 移动端App（最终形态）
├── 外接音乐源（实验性）
└── 可穿戴设备联动（远期）
```

### 为什么先做网页不做小程序？

小程序的音频后台播放有平台限制，网页端反而更自由，适合快速验证核心体验。
